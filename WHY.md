# 為什麼做這個專案

> 這份文件記錄 PostgreSQL Gym 的設計動機與當初的取捨，避免理由只活在記憶裡。
> 實作細節看 [README.md](README.md)。

## 一句話定位

這是 **Node.js 課程 PostgreSQL 單元的線上練習場** —— 讓學生在瀏覽器裡「真的動手寫 SQL」、邊做邊驗收，而不是看影片或抄答案。定位是**學生自己動手的任務（task）系統**，資料情境用健身房（LiveFit / gym domain）。

## 核心動機與取捨

| 動機 | 當初的決定 |
|------|-----------|
| **貼著既有影音課程走** | 前半段對著現有 SQL 影片章節（資料庫簡介 → 建表 → 語法 → JOIN → GROUP BY → 子查詢），先建手感再進難題；每課只展開當下相關的影片模組，不一次倒整份課綱 |
| **跟 Node.js 課最終專案接得起來** | 練習用的 `credit_packages` 資料表刻意對齊 Node.js 課 week4 的 `CreditPackage` entity（`name` / `credit_amount` / `price` / `created_at`），學生練的資料形狀＝之後課程要寫的 |
| **用「真的 PostgreSQL」而非 SQLite** | 用 PGlite 在瀏覽器跑真 Postgres 語意，學生不用裝環境 |
| **驗收看結果、不看字串** | 用資料庫狀態 / 查詢結果判對錯，不比對 SQL 字串；但可針對每題要求一定要用 `WHERE` / `ORDER BY` / `LIMIT` 等學習目標 |
| **降焦慮、循序漸進** | 用海姐（coach-hex）陪跑，先安撫情緒再給小步驟；新手只先開 `members` 單表，不一次倒一堆 |
| **後段做情境題** | 規劃 SQLNoir 風格的「事故調查」情境關卡，放在新手基礎之後（例：05 Northwind Lite 電商訂單巡查、11 事故室 SQLNoir 風格後端調查） |

## 重要的方向轉變：這是「任務」不是「練習題」

開發中途確立了一個關鍵原則：**這是一份任務（task），不要讓學生直接拿到答案。**

因此介面從「練習題 + 參考答案」改成任務框架：

- 中央改顯示「本章任務 / 任務目標 / 任務步驟」
- Hint 3 從完整解答改成「驗收前檢查清單」
- 參考 SQL 只有在通過該任務後才出現
- 起始 SQL 調整成不會剛好就是答案

## 參考來源（設計靈感）

- **SQLBolt / PGExercises**：前半段的短教學 + 漸進式新手練習
- **SQLNoir**：後半段的情境 / 事故調查任務
